
# Desktop Build with Tauri

In addition to the web and PWA versions, Lunara was also packaged as a Windows desktop application using Tauri.

The goal of this stage was to take the product beyond the browser, allowing users to install Lunara on their computer and access the application as a standalone app.

This reinforces the project’s proposal as a complete management solution, not only as a published web application.

## Why Tauri

Tauri was chosen because it allows a web application to be transformed into a desktop app efficiently.

Since Lunara had already been developed with React, TypeScript, and Vite, Tauri made it possible to reuse the existing front-end and generate a desktop version without rewriting the application in another technology.

Another important point is that Tauri generates lighter applications compared to solutions based on Electron. For Lunara, this made sense because the goal was to create a functional desktop version with good performance while taking advantage of the codebase that had already been built.

## Structure Used

The main project remains a Vite application.

The web build is generated in the following folder:

```txt
dist/
```

In `tauri.conf.json`, Tauri was configured to use this build:

```json
"build": {
  "frontendDist": "../dist",
  "devUrl": "http://localhost:5173",
  "beforeDevCommand": "pnpm dev",
  "beforeBuildCommand": "pnpm build"
}
```

This allows Tauri to use the final files generated by Vite to create the desktop application.

During development, it was also possible to run the app in desktop mode using the local Vite server through the `devUrl`.

## App Configuration

The desktop app was configured with a name, identifier, and specific window dimensions:

```json
"productName": "Lunara",
"identifier": "com.usuario.lunara"
```

The main window was configured like this:

```json
"width": 1200,
"height": 800,
"minWidth": 390,
"minHeight": 640
```

The initial width and height were defined to provide a comfortable desktop experience. The minimum size helps keep the interface usable even when the window is resized.

## Icons

The desktop app icons were generated from Lunara’s main icon.

The command used was:

```txt
pnpm tauri icon public/favicon-maior.png
```

This command generates the required formats for different platforms, including:

```txt
32x32.png
128x128.png
128x128@2x.png
icon.ico
icon.icns
```

For Windows, the main file is `icon.ico`, which is used by the application and the installer.

## CSP and Firebase

Since Lunara uses Firebase Authentication and Firestore, it was necessary to configure Tauri’s Content Security Policy to allow connections to the required domains.

This configuration is located in `app.security.csp`.

Firebase and Google-related domains were allowed, such as:

```txt
googleapis.com
firebaseio.com
firebaseapp.com
firestore.googleapis.com
identitytoolkit.googleapis.com
securetoken.googleapis.com
firebaseinstallations.googleapis.com
```

This configuration is important because the desktop app needs to authenticate users and access the database normally.

Without a proper CSP, the desktop window could open, but some external requests could be blocked.

## Windows Installer Build

The Windows installer was generated with:

```txt
pnpm tauri build
```

This command runs several steps:

* generates the Vite build;
* compiles the Tauri application;
* packages the app;
* generates the Windows installer.

The final installer was created in:

```txt
src-tauri/target/release/bundle/nsis/
```

The generated file was:

```txt
Lunara_0.1.0_x64-setup.exe
```

## Note About Digital Signing

Since the installer does not have a digital signature yet, Windows may display a security warning when opening the file.

This behavior is common for independent desktop applications that do not yet have a signing certificate.

## Result

With the Tauri configuration, Lunara now includes:

* web version;
* PWA version;
* Windows desktop version;
* `.exe` installer;
* custom icons;
* Firebase integration also working in the desktop environment.

This stage strengthens the project as a product because it shows that the application was designed not only to run in the browser, but also to be distributed as desktop software.
