
# Firebase and Data Security

Lunara uses Firebase for authentication and data persistence.

The choice of Firebase had two main reasons. First, I wanted to get more familiar with the tool and better understand authentication, databases, and security rules without having to build a custom API from the start, as I usually do. Second, Firebase fit the project well because it makes it possible to create a solid foundation for a front-end application without adding too much backend complexity.

In Lunara, Firebase was not used only as an extra feature. It is part of the product’s core architecture.

## Firebase Authentication

Firebase Authentication was used to allow each user to create an account and access the app with email and password.

This part handles user identification inside the system. In other words, the app can know who is logged in and use that information to control the experience and data access.

However, one important point is that authentication alone does not guarantee complete security.

Having a working login means the user has been identified. But to protect the data, it is also necessary to make sure that each user can only access information that belongs to them.

For that reason, Lunara also uses a user-based data structure and Firestore security rules.

## Cloud Firestore

Cloud Firestore was used to store the main information of the app:

* clients;
* projects;
* tasks;
* financial transactions.

This data is essential to the Lunara experience because it represents the user’s professional routine inside the application.

Since the project involves client information, deadlines, financial values, and work organization, correctly separating this data was an important part of the architecture.

## User-Based Data Isolation

Lunara is a multi-user application. This means that each person can create an account and use the app with their own data.

For this to work correctly, the data cannot be mixed in global collections. Each user needs to have their own space inside the database.

The final structure was organized like this:

```txt
users/{uid}/clients
users/{uid}/projects
users/{uid}/tasks
users/{uid}/transactions
```

The `uid` is the unique identifier generated by Firebase Authentication for each account.

In practice, this means the data is separated by user.

Example:

```txt
users/abc123/clients
```

and

```txt
users/xyz789/clients
```

are different spaces inside the database.

With this structure, one person can register their own clients, projects, tasks, and transactions without that data appearing for another user.

## Firestore Rules

In addition to the structure in the code, Firestore rules were also configured to prevent unauthorized access.

The rules allow an authenticated user to access only the documents inside their own `uid`.

```txt
rules_version = '2';

service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId}/{collectionName}/{documentId} {
      allow read, create, update, delete: if request.auth != null
        && request.auth.uid == userId;
    }

    match /{document=**} {
      allow read, write: if false;
    }
  }
}
```

This configuration guarantees two things:

1. only authenticated users can access data;
2. each user can only access data linked to their own `uid`.

The final rule blocks any read or write operation outside the allowed structure.

## Why This Decision Matters

Lunara handles data that is part of the user’s professional routine.

Clients, projects, tasks, deadlines, and financial values are information that need to be protected. For this reason, user-based data isolation is not just a technical detail. It is an essential decision for the reliability of the product.

This adjustment also shows an important difference between creating a functional interface and building an application that is responsible for handling user data.

## What Was Validated

After adjusting the structure, data isolation was tested with different accounts.

The validated flow was:

* create an account;
* register data in that account;
* sign out;
* log in with another account;
* confirm that the data from the first account does not appear;
* create data in the second account;
* return to the first account;
* confirm that the data remains separated.

This test confirmed that each user has their own dashboard and their own information.

## Relationship Between Auth, Firestore, and Rules

One of the most important learnings from this stage was understanding that Firebase Authentication, Firestore, and Firestore Rules have different roles.

```txt
Firebase Auth identifies the user.
Firestore stores the data.
Firestore Rules control access.
```

These three parts need to work together.

In Lunara, the authenticated user is used as the base to define where data will be stored and which information can be accessed.

This makes the application more secure, more organized, and better suited for a real usage scenario.

## Result

With authentication, the user-based structure, and security rules, Lunara now has a more consistent and reliable data foundation.

This stage was essential to strengthen the project as a product, because it ensures that each user has an individual experience inside the application, with their data protected and separated from everyone else’s.
